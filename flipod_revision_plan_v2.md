# Flipod 改版方案 v2

> 来源：2026-04-26 与 AI 的产品 brainstorming（多轮校正后定稿）
> 用途：内部对齐 + 后续 Claude Code handoff 的输入
> v1 已废弃 —— 基于错误的 phase 假设（误以为 5 phase / phase 1 = 拆词）

---

## 背景

**当前症状**
- UI 视觉密度高、选项多、文字多
- "纯听 tab" 和 "练习 tab" 两个并行入口让用户在第一屏就要做模式选择，是认知负担
- 收到两类内部反馈：CEO 说 UI 太复杂、同事说切片太难

**对内部反馈的判定**
- CEO 和同事都不是目标用户，都是 lousy PM。反馈是 stakeholder 政治不是用户研究
- 字面采纳会让产品偏离定位（UI 简到 CEO 满意 + 难度降到同事满意 = 一个面向初学者的简单 app）
- 但反馈触发的两个**根因洞察**是真的，需要解决：
  - 首屏 30 秒不能 grok 产品（不是按钮问题）
  - 用户面对难内容时归因到"我笨"而非"内容难"（不是难度问题）

**不变量（任何改版决策的边界）**
- 目标用户：中级听力 + 愿意听播客
- 定位：**刷感外壳 + 教学专业引擎 + 陪伴型内部基调**
- 内容客观难度不降
- "陪伴"不进外部文案（只进 microcopy 基调）

---

## 实际的 phase 结构（基于 practice-v2-demo.html 实际代码）

| 阶段 | 当前实现 | 听力训练理论 |
|------|---------|------------|
| **0 · 预听** | 看标题 + 关键词 chips，10-15s 猜内容 + 预测题 | Schema activation / top-down 预测 |
| **1 · 把意思（Gist）** | 中文字幕盲听一遍，不能暂停/回退，主旨题 | Gist comprehension under pressure |
| **2 · 解码（Decode）** | 英文字幕（可点显中文），可暂停 + 后退 3s | Intensive decoding / 词级理解 |
| **3 · 渐隐** | 实词消失（▁），保留 B2+ 高级词 + 功能词 | Cloze listening / bottom-up sound-to-word recovery |
| **4 · 检验 + 归因** | 无字幕完整听 → 6 类卡点 chip → 选 unknown 进 step2 标具体词 | Metacognitive listening strategy |
| **5 · 词汇复习** | 从 step2 标的 C1+ 词复习 | Vocabulary consolidation |

引擎本身（教学链路）非常专业，**v2 不动这套引擎**。要动的是外壳怎么把它包成"看起来轻"的东西。

## 改什么（5 个核心动作）

### 1. 智能体方案 = C2（对话主导 + 保留刷流入口）

**默认状态**：对话流（智能体推一条聊一条）
**保留**：刷流入口（用户主动「我自己挑」时进入 feed 卡片流）
**切换**：显式，用户主动触发，不是混合 UI

**首次打开（30 秒 onboarding）**

**【2026-04-26 重大修订】取消"推荐 vs 自挑"二选一**

之前设计是 onboarding 让用户在「试试推荐」和「自己挑」之间二选一。后续讨论发现这个二分是冗余的：

- 两条路径的练习层最终都走相同的 6 phase 流程
- 两条路径的内容池也相同（智能体推的 = 刷流候选）
- 让用户在第一秒做选择是认知冗余，应该让用户**直接开始**

新方案：**默认即无限流主线，智能体推一条用户练一条**。"自己浏览候选"的需求收到侧边菜单（详见 §6 侧边菜单设计）。

---

**首次 OB 流程（应用第一次打开 · 教学性介绍，不让用户做决策）**

```
屏幕中央：音波形象 + 对话气泡（自动连续 + 关键节点 chip）

段落 1（自动播）：「想听一条吗？还是我来推荐？」（这一句是占位，确切措辞细节后定）
  ↓ 0.6s 后自动进段落 2
段落 2（自动播）：「我先讲一下流程 —— 进入 clip 之前会看到一个 priming 界面，让你预热」
  ↓ 自动
段落 3（自动播）：「然后是中文 → 英文 → 挖空 → 盲听 → 总结，中间会问几道题」
  ↓ 出现 chip：「明白」/「再说一遍」← 关键节点 chip
段落 4（自动播）：「碰到不会的词，可以直接点 → 加进生词本」
  ↓ 自动
段落 5（自动播）：「我先给你来一条试试」
  ↓ 自动进入第一条 clip 的 priming
```

**第一条完成后智能体二次介入**（不立刻打断，**在下一条 priming 时插入对话**）

```
clip 完成（phase 5 结束）→ 自动进入下一条 priming（倒数 ring 启动）
  → 智能体在 priming 界面顶部出现，对话气泡：
    「右上角的菜单 —— 里面有生词本、收藏、爱心」
    ↓ chip：「看看 / 等会再说」
    →（看看）打开侧边菜单 → 介绍各项功能：
      - 生词本：你点过的词都在这，**会影响后续推荐**
      - 收藏：喜欢的 clip 留这
      - 爱心：爱心一条之后，可以单独纯听这条（不练）
      - 兴趣：换学习方向
    →（等会再说）智能体退场，priming 倒数继续
```

**日常循环（OB 之后所有时刻）**

```
clip 完成 → priming（带倒数 ring，3 秒后自动进下一条）
  → 用户可以：上滑换条 / 长按暂停 / 等倒数自动进
  → 进下一条 phase 1 中文盲听
  → ... 6 phase 流程 ...
  → 完成 → priming → 下一条
```

**核心循环单元 = priming + clip**，不回首屏，不做决策中断。

---

**为什么 C2 → C2.1（取消二选一）**
- 之前 C2 的"对话主导 + 刷流入口"在 onboarding 上做成了二选一，反而加重了认知负担
- C2.1 把"刷流自挑"的需求收到侧边菜单的"看看有什么"入口（轻量浏览候选），主流程纯线性无限流
- 既保留 C2 的两种用户心情兼顾（被推 / 主动挑），又消除二选一的决策疲劳
- 主流程极简（一条接一条），但用户随时可从菜单进入主动浏览模式

**陪伴型基调**
- 不进 AppStore 描述、不进首屏标题
- 进每一句 microcopy（答错时、长时间未访问、过渡话术、智能体讲解）

### 2. 砍 Phase 0 的预测题；Phase 1 同时承担教学 + 试听筛选

**新流程**
```
用户从对话/刷流选 clip
  → 直接进 phase 1（中文字幕盲听 · 全程不能暂停/回退）
  → 0-30 秒：× = 取消（弹确认），不算开始
  → 30 秒后：底部出现「先到这吧」按钮 = partial_listen 退出
  → 听完整段：出主旨题
  → 三个选项：
      [选项 A]→ 进 phase 2（继续精听）= committed
      [选项 B]→ 进 phase 2（继续精听）= committed
      「今天就到这吧」→ 退出 = completed_phase1
  → 选继续 → phase 2-5 走完整流程
```

**为什么这样设计**
- 决策点从"开始之前"挪到"已经听了一遍中文之后" —— 承诺成本最低（直接开听），决策建立在沉没成本上
- Phase 1 同时承担教学（gist comprehension）和试听（决定要不要继续），**消除"试听破坏 novelty"的矛盾** —— 用户听的就是真正的练习内容
- **主旨题既测 gist 又是 commitment 动作**，三个选项里嵌入"退出" CTA，让退出成本可控但不无痛

**30 秒承诺门槛**（2026-04-26 增补，让 onboarding 文案"想听 30 秒？"真实成立）
- 0-30 秒：不可暂停 / 不可退出（× = 取消语义，弹确认）
- 30 秒后：底部出现「先到这吧 →」按钮（中性灰），按 = partial_listen
- 30 秒倒计时**不可见**（按钮自然出现，不让用户盯倒计时破坏沉浸）
- 全程不允许"暂停"——只允许"退出"。30 秒后多了"退出"权利，不多"暂停"权利
- 设计意图：把"30 秒"从早期方案的 clip 长度（已废弃），改造成产品对用户的真实承诺。"想听 30 秒？" = "你只需要承诺 30 秒注意力，之后随时退"

**四种退出/完成的数据语义**
- `cancelled`（0-30 秒按 ×）：不算 session，不进推荐信号
- `partial_listen`（30 秒后按「先到这吧」）：算一次接触，进推荐信号（"用户对 X 类内容听了 30+ 秒就走"）
- `completed_phase1`（听完后选「今天就到这吧」）：最浅完成
- `committed`（听完进 phase 2）：强信号

**教学损失评估**
- 失去 phase 0 的 active prediction（强迫预测的深度加工）
- 但中文盲听本身是 standard extensive listening，是 schema activation 的 passive 形式，仍成立
- 听力教学理论里 pre-listening 不必须是预测题（可以是标题、图、词表、听一遍），砍预测题不算大伤

### 3. 容器复用 —— 纯听 tab 升级为完整流程入口

**不是"砍纯听 tab"，是把它升级为所有 phase 的容器**：

- 容器：`feed-card-screen`（全屏 scroll-snap 卡片流，上下滑切换 clip）—— 这就是"刷"的物理形态
- Phase 切换 = 卡片内功能变化（字幕模式 + 控件可用性），**不是路由跳转**
- 同一个容器承载 phase 1-5

**功能映射**

| Phase | 字幕 | 控件 | 备注 |
|-------|------|------|------|
| 1 | 中文 | ❌暂停 ❌回退 | 划走 = 终止 |
| 2 | 英文（可点显中文） | ✅暂停 ✅后退 3s | 不给快进 |
| 3 | 渐隐（实词▁，B2+ 词 + 功能词保留） | ✅暂停 ✅后退 3s | |
| 4 | 无字幕 | ❌暂停 ❌回退 | + 归因 chip 网格 |
| 5 | 词汇卡片层 | — | 从 step2 标记词中筛 C1+ |

**好处**
- UI 一致性高，用户不在不同 phase 间感受到 context 切换
- 工程量降低 —— 一套容器复用，不用维护两套 UI（练习 tab + 纯听 tab）
- "刷"和"精听"在物理形态上统一 —— 都是同一种全屏卡片，只是卡片内容不同

### 4. Priming 过渡界面（复用 demo/priming-v4.html）

**作用**：每条 clip 之间的过渡缓冲，是"无限流核心循环单元"的前半部分。

**复用现有实现**：`demo/priming-v4.html` 已经做了：
- 标题 + summary + 关键词 chips
- 中央倒数 ring（默认 3 秒后自动进 clip）
- 上滑切换下一条
- 长按暂停

**核心循环**：`priming → clip (6 phase) → 下一条 priming → 下一条 clip → ...`
- 倒数 ring 让用户有心理缓冲（不是被推进 clip 的，是自己倒数后进的）
- 上滑换条让用户在 priming 阶段就能"换一条"，不必进 clip 才能拒绝

**与智能体的配合**：
- 默认 priming = 纯倒数 + 内容预热（无智能体）
- **特殊 priming**（首次 OB 后第一条完成时）= priming 顶部出现智能体气泡，介绍侧边菜单
- 第 N 次特殊 priming = 智能体阶段性 review（"你最近卡过 X 5 次，要不要试试 Y 类内容？"）

### 5. 刷流卡片简化（移除 phase 0 的杂物）

原 phase 0 把标题/来源/关键词/选择题全堆在一个面板上。新设计**拆开**：

**刷流里的卡片**（用户在 feed 卡片流里看到的）
- 节目名（信任来源 —— The Daily / Hard Fork / The Moth ...）
- 这一段的标题
- 嘉宾（如有）
- 时长
- **一句 hook**（智能体生成，制造悬念不剧透答案，例："为什么 OpenAI 突然让模型变笨了？"）
- 难度暗示标签
- 推荐理由小标签（智能体生成，例："因为你上次卡过 weak forms"）

**关键词 chips → 移到 phase 1 中文字幕上方做预热**（不在卡片上）

**点击「精听」** → 直接进 phase 1（中文字幕盲听）

## 6. 侧边菜单设计（承接非主流程功能）

主流程极简（priming + clip 无限流），侧边菜单容纳所有"二级功能"。汉堡按钮固定在 priming 和 clip 的右上角，随时可呼出。

**菜单项**：

| 项目 | 用途 | 解锁/触发 |
|------|------|----------|
| **生词本** | 用户点过收藏的词，按时间倒序 | 首次有词时智能体提示"会影响后续推荐" |
| **收藏** | 用户收藏的 clip | 任意 clip 中的收藏按钮触发 |
| **爱心** | 用户标爱心的 clip | 任意 clip 中的爱心按钮触发 |
| **看看有什么** | 主动浏览候选 clip 列表（轻量 feed 卡片视图） | 默认可见 |
| **学习兴趣** | 选择 / 更换学习方向（如：科技、故事、新闻、文化）| 默认可见 |
| **设置** | 主题、左手模式、重置等 | 默认可见 |

**关键设计决策**：

- **「爱心」的功能性意义**：不只是 vanity metric。爱心一条之后，**这条 clip 解锁纯听模式**（用户可以反复纯听欣赏，不必再走精听）。这给了"纯听 tab"一个克制的存在形式 —— 不是默认入口，是"已精听过的内容才能纯听"
- **「生词本」是推荐的真实输入**：词进生词本后进入推荐算法的负反馈信号（"用户卡过这词族 → 后续推同类内容时给 framing 提示"）。这要在智能体首次提到生词本时讲清楚（"会影响后续推荐"），让用户的每一次点词都有意义感
- **「看看有什么」承接被砍掉的"自挑"二选一**：用户偶尔想"看看新东西"时进入，是轻量浏览候选 feed 卡片视图。不是默认入口，但随时可达
- **「学习兴趣」的修改时机**：除了用户主动从菜单进入，智能体也可以在阶段性 review 时主动建议（"你最近常听科技类，要不要加点别的？"）

## 数据结构

**每条 clip 需要附带的元数据**
```json
{
  "card_hook": "刷流卡片上的悬念句（智能体生成，不剧透）",
  "card_difficulty_hint": "难度暗示标签（如：中等 / 偏快 / 含口音）",
  "recommendation_reason": "推荐理由（基于用户上次表现）",
  "keywords": "关键词数组（用在 phase 1 字幕上方，不在卡片上）",
  "framing_hint": "phase 1 进入前的一句话提示（如：抓主旨即可 / 试着跟读）"
}
```

**每个用户需要追踪的状态**
```json
{
  "is_first_visit": "是否首次（决定走 onboarding 闭环）",
  "last_clip_attribution": "上次精听的 6 类归因（unknown/linking/weak/speed/accent/other）",
  "chat_history": "对话流的历史（用于推荐 + 续上次）",
  "swipe_signals": "刷流里的划走/停留/标记（用于偏好调整）",
  "stuck_patterns": "重复卡住的语音/语法模式（用于推荐解释）"
}
```

## 不做清单（明确边界）

- ❌ 不降低 clip 内容客观难度
- ❌ 不为初学者设计任何功能
- ❌ 不做"试听"功能（破坏 novelty）
- ❌ 不在 phase 1 之前加任何预热步骤（包括原 phase 0 的预测题）
- ❌ 不让对话和刷流混合呈现（必须显式切换）
- ❌ 不在外部文案 / AppStore 描述里出现"陪伴"二字
- ❌ 不立刻完善后端推荐算法（等真实用户信号）
- ❌ 不为响应 lousy PM 的具体 UI 建议而砍 phase 数量
- ❌ 不在没有真实用户数据前响应 stakeholder 的具体产品建议

## 自检清单（做之前 + 做之后都过一遍）

1. **首屏测试**：一个完全不知道 Flipod 是什么的人，30 秒内能 grok 产品价值吗？
2. **智能体台词**：这句话是在帮 grok / framing / 归因管理？还是只是替代了一个按钮？后者要删
3. **专业引擎**：5 个 phase（1-5）每一阶段对应的听力训练目标，能用一句话讲清吗？
4. **定位边界**：这个决策偏离"中级用户定位"了吗？
5. **基调一致性**：这条 microcopy 是温和不评判？还是滑向了督促/打卡/KPI？
6. **容器一致性**：这个新增的功能是放在 feed-card-screen 容器内（功能解锁/锁定）还是又开了一个新 UI？后者要警惕

## 验证计划（最重要 —— 不做这步前面都白搭）

**优先级**：找真实用户 > 完善智能体设计 > 回应 stakeholder 反馈

**具体步骤**

1. **招募 5-10 个真实目标用户**
   - 画像：中级听力（雅思 6-6.5 / CET6 通过 / 能听懂 60-80% 真实播客）+ 愿意听播客
   - 来源：身边的人、播客听众社群、小红书相关话题、播客类播主的粉丝群
   - 不要：英语很差的人（不是目标用户，反馈会误导）

2. **测试材料：MVP demo**
   - 不需要全功能 —— onboarding 30 秒闭环 + phase 1 中文盲听 + 主旨题 + phase 2 + 1-3 条 clip 就够
   - 重点测产品心智，不是功能完备

3. **观察三个时间维度**
   - 30 秒后：能 grok 产品在干嘛吗？
   - 3 分钟后：完成第一个完整 clip 后，是放松还是挫败？
   - 3 天后：还会主动再打开吗？

4. **每周复盘**
   - 真实用户信号 vs lousy PM 反馈的偏差
   - 用真实信号校准产品方向，不用 PM 反馈

**搁置**：在拿到至少 5 个目标用户的真实使用数据之前
- 不响应 CEO 的具体 UI 建议（用"给他能 demo 的 30 秒叙事"满足他的真实 JTBD）
- 不响应同事的具体难度建议
- 不做大规模重构

## 待写进 PRD 的修订（2026-04-26 brainstorm 新增）

### Phase 4 step 2「生词卡点」交互改造

**用户原话**："如果是因为生词的话不要给几个选择让用户挑了，让他自己点击词汇然后收录进生词本里作为推荐逻辑的参考之一"

**与 PRD 现状的差异**：
- PRD §L4：当前规则是 "归因含生词即触发 phase 5，空 vocab_picked 时用系统识别的 C1+ 词兜底"——本质是**系统提供选项让用户在四选一中挑词**
- PRD §D9：渐隐字幕（phase 3）有长按菜单"我认识/不认识"——是被动反馈，不是主动收录
- **新方向**：用户在精听过程中（phase 2 / 3 的 visible 词）**主动点击不认识的词** → 直接收录进生词本 → 作为后续推荐逻辑的参考

**待 PRD 决议**：
1. 取消 phase 4 step 2 的"四选一挑词"，改成 phase 2/3 字幕中的"点词 = 收录"
2. 收录后的词进入 `vocab_inbox`（已有字段），同时进入推荐算法的负反馈信号（"用户卡过这个词的近似词形/词族 → 后续推同类内容时给 framing 提示"）
3. Phase 5 词汇复习的输入源：从 `vocab_inbox` 中筛 C1+ 且最近 N 天加入的词（不再依赖 step 2 的 `vocab_picked` 字段）
4. Phase 4 step 2 是否完全砍掉？或保留为兜底（用户没在 phase 2/3 主动点词，归因又选了"生词"时启用）—— 待定
